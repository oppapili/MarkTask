## Q1. 解決したい中核課題は何ですか？

## Q5. 重視する利用形態はどれですか？（複数選択可、重視する順で）

- A. CLI（コマンドラインでの操作）
- B. GUI（デスクトップ / Web アプリ）
- C. API（外部連携・自動化）
- D. エディタ連携（VS Code 等でファイルをそのまま編集できれば十分）
- E. すべて同等に重視
- X. その他（自由記述）

[Answer]: E → Q8 で改訂。**独自GUIは重要でない**。MVP の主要インタフェースは (1) **CLI**、(2) **Obsidian互換のMarkdownファイル形式**（frontmatter＋wikilink `[[...]]` でタスクのメモとナレッジを相互リンク）、(3) **MCP（Model Context Protocol）** — 生成AIが MarkTask をツールとして使えるように備えたい。API相当は CLI/MCP で充足。

## Q7. MVP で絶対に外せない機能はどれですか？（複数選択可）

- A. タスクの CRUD（作成 / 閲覧 / 更新 / 削除）＝1タスク1Markdownファイル
- B. タスク一覧・フィルタ・検索
- C. 状態管理（TODO / 進行中 / 完了 など）
- D. メタデータ（frontmatter による期限・優先度・タグ）
- E. 上記すべて
- X. その他（自由記述）

[Answer]: E — 加えて、**recurrence（繰り返しタスク）の管理** も MVP に含める。方針: **frontmatter に繰り返しルールを記述**して単一ファイルで管理する（完了ごとに新ファイルを生成する方式は 1ファイル1タスクの命名で衝突しやすく、過去の蓄積情報を都度参照する運用にも合わないため避ける）。

## Q8. 生成前の最終確認（解釈とスコープ優先度）
以下の解釈で intent-statement / stakeholder-map を生成してよいか確認してください。

- 解釈1: 中核は「1タスク=1Markdownファイル」＋ Obsidian vault 親和（wikilink でタスクとナレッジを相互リンク）。独自GUIを作り込むより、CLI ＋ Obsidian互換ファイル形式（frontmatter＋wikilink）を中核とし、GUI/API は補助的位置づけ。
- 解釈2: データは ローカルのMarkdownファイル群が Single Source of Truth。クラウド同期なし（Git併用は任意）。
- 解釈3: recurrence を MVP に含む。Markdownファーストとの表現方法は後段（scope-definition / requirements）で具体化。
- スコープ優先度: MVPの核を「CLI＋ファイル形式＋CRUD/状態/メタデータ/recurrence」に置き、一覧・検索やObsidian連携の作り込みは段階的に進める、という優先度でよいか。

- A. この解釈・優先度でOK。生成してよい
- B. 解釈はOKだが、全機能を等しくMVP重視にしたい（段階化しない）
- X. その他（修正点を指定）

[Answer]: X — 修正: (1) 独自GUIは重要でない。MVPは「生成AIでも使える CLI ＋ Obsidian互換ファイル形式」でOK。加えて **MCP をインタフェースとして備えたい**。(2) OK（ローカルのみ）。(3) OK、ただし **frontmatter に繰り返しルールを書く方向**を志向（過去に蓄積した情報を都度参照したい／1ファイル1タスクの命名衝突を避けるため、完了時に新ファイル生成する方式は採らない）。
