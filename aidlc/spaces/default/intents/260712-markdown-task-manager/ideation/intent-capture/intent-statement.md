# Intent Statement — MarkTask

> Ideation フェーズ / intent-capture ステージの成果物。問題と価値の枠組みを定義する（実装・アーキテクチャの詳細は後段の Inception 以降で扱う）。
> Scope: **mvp** ／ 対象ユーザー: 個人（自分）

## Problem Statement（解決したい課題）

既存のタスク管理ツールでは、**タスクの詳細を Markdown で自由に（リッチに）記述できない**。多くのツールが独自の入力欄・独自形式に閉じており、思考やメモを本文としてそのまま書き残しにくい。結果として「タスク」と「そのタスクにまつわるメモ・ナレッジ」が分断される。

MarkTask は「**1タスク = 1 Markdown ファイル**」という単純な原則で、タスクそのものを自由記述の Markdown ドキュメントとして扱う。これにより、タスク本文に思考・手順・参照リンクを自然に書き込め、ファイルはユーザーの資産として長期に残る。

ただし、Markdown（プレーンなファイル）ベースの管理は **繰り返し（recurrence）タスクの扱いが弱い** という既知の弱点を持つ。MarkTask はこの弱点を正面から補うことを課題に含める。

## Target Customer（対象ユーザーと便益）

- **主対象**: 自分自身（パーソナルユース）。Git / Markdown 中心のワークフローを持ち、テキストで素早く記述したい個人。
- **得られる便益**:
  - タスク＝自由記述の Markdown なので、思考やメモをタスクに直接残せる。
  - ファイルはプレーンな Markdown のまま。ツールを離れても（エディタ／Obsidian／git で）読め、ロックインが無い。
  - **Obsidian 連携**: タスクファイルを Obsidian の vault 内で自然に扱え、wikilink（`[[...]]`）でタスクのメモを既存のナレッジノートと相互に紐づけられる。
  - **生成AI からの利用**: MCP（Model Context Protocol）インタフェースを通じ、AI エージェントが MarkTask をツールとして操作できる。

## Success Metrics（成功の測定可能な指標）

MVP が成功と言える状態（測定可能な形で定義）:

1. **完全移行**: 自分の日常タスク管理を MarkTask に移行でき、他のタスク管理ツールの併用がゼロになる。
2. **ロックインゼロ**: すべてのタスクがプレーンな Markdown ファイルとして残り、MarkTask を介さずに（Obsidian／任意のエディタ／git）100% 読める。
3. **CLI 完結**: タスクの主要操作（作成・状態変更・一覧/検索・完了・繰り返し設定）が CLI だけで完結する（GUI 不要）。
4. **AI 操作可能**: 生成AI が MCP 経由でタスクの作成・参照・更新・完了を実行できる（少なくとも主要操作を MCP ツールとして公開）。
5. **繰り返し対応**: 繰り返しタスクが破綻なく管理できる（次回発生日の算出・消化が、ファイル名衝突や履歴喪失を起こさずに行える）。

## Initiative Trigger（なぜ今か）

- 既存ツールへの不満が蓄積している（Markdown 自由記述ができない、独自形式への不信）。
- 併せて、**学習・技術検証**を兼ねて自作したい（Markdown ファースト設計、Obsidian 親和、MCP による AI 連携という新しい題材を自分の手で確かめたい）。

## Initial Scope Signal（初期スコープの手がかり）

- **スコープ**: `mvp`（作り切って自分の実運用に載せる。運用フェーズ＝本番デプロイ/監視等は今回のスコープ外）。
- **主要インタフェース（MVP に含む）**:
  - CLI（第一級のインタフェース）
  - Obsidian 互換の Markdown ファイル形式（frontmatter ＋ wikilink）
  - MCP サーバ（生成AI 向けインタフェース）
- **中核機能（MVP に含む）**:
  - タスクの CRUD（1タスク=1Markdownファイル）
  - 一覧・フィルタ・検索
  - 状態管理（例: TODO / 進行中 / 完了）
  - frontmatter によるメタデータ（期限・優先度・タグ 等）
  - **繰り返し（recurrence）管理** — frontmatter に繰り返しルールを記述し、単一ファイルで管理する方向（完了ごとの新ファイル生成は、命名衝突と履歴分断を避けるため採らない）。
- **明示的に MVP 対象外**:
  - 独自 GUI（デスクトップ/Webアプリの作り込み）— Obsidian と CLI で代替。
  - クラウド同期 — データはローカルの Markdown ファイル群が Single Source of Truth（同期は各自の運用／git 等に委ねる）。
- **優先度シグナル**: 核（CLI ＋ ファイル形式 ＋ CRUD/状態/メタデータ/recurrence ＋ MCP）を先に固め、一覧・検索や Obsidian 連携の作り込みは段階的に。
- **要確認事項（後段へ引き継ぎ）**: 繰り返しルールの frontmatter スキーマ、MCP で公開する操作範囲は scope-definition / requirements-analysis / application-design で具体化する。
